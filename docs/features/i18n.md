# Internationalization (i18n)

MedBuddy uses next-intl for internationalization with Traditional Chinese (zh-TW) as the default locale and English (en) as the secondary locale. The locale prefix strategy is "as-needed", meaning zh-TW URLs have no prefix while English URLs are prefixed with `/en`.

## Architecture

### Routing Configuration

`src/i18n/routing.ts` defines the routing setup:

```typescript
export const routing = defineRouting({
  locales: ["zh-TW", "en"],
  defaultLocale: "zh-TW",
  localePrefix: "as-needed",
})
```

This means:
- `/home` serves the zh-TW version (no prefix for the default locale).
- `/en/home` serves the English version.
- The module also exports locale-aware navigation utilities: `Link`, `redirect`, `usePathname`, `useRouter`, `getPathname`.

### Request Configuration

`src/i18n/request.ts` is the server-side configuration that loads the correct message file based on the request locale. If the locale is invalid or missing, it falls back to the default (`zh-TW`). Messages are loaded via dynamic import from the `messages/` directory.

### Next.js Integration

`next.config.ts` wraps the Next.js config with `createNextIntlPlugin`, pointing to `./src/i18n/request.ts` as the request configuration file.

### Middleware

The middleware in `src/proxy.ts` runs `next-intl`'s `createMiddleware(routing)` first for locale detection and prefix handling. Subsequent auth/onboarding checks strip the locale prefix before matching protected routes, so route protection works regardless of locale.

### Translation Files

Translation files live in the `messages/` directory:

- `messages/zh-TW.json` -- Traditional Chinese (default)
- `messages/en.json` -- English

Both files share the same key structure. Top-level namespaces include:
- `common` -- Shared UI strings (loading, save, cancel, etc.)
- `nav` -- Bottom navigation labels
- `auth` / `authPages` -- Authentication forms
- `landing` -- Landing page content
- `onboarding` -- Onboarding wizard
- `home` -- Home/today page
- `medications` -- Medication management (list, add, scan, detail, interactions)
- `chat` -- Chat interface
- `caretaker` -- Caretaker dashboard
- `profile` -- Profile settings
- `timeSlots` -- Morning, afternoon, evening, bedtime
- `status` -- Pending, taken, missed, skipped
- `errors` -- Error page
- `notFound` -- 404 page

### Usage in Components

- **Client components**: `useTranslations("namespace")` from `next-intl`.
- **Locale detection**: `useLocale()` from `next-intl`.
- **Locale-aware links**: Import `Link` from `@/i18n/routing` instead of `next/link`.
- **Locale-aware router**: Import `useRouter` from `@/i18n/routing`.

The user's preferred locale is also stored in the `user` table (`locale` column) and used for AI-generated content (medication purposes, chat responses). It can be updated via `PATCH /api/profile` with `{ locale: "en" }` or `{ locale: "zh-TW" }`.

## Key Files

| File | Purpose |
|------|---------|
| `src/i18n/routing.ts` | Locale routing config and navigation exports |
| `src/i18n/request.ts` | Server-side message loading |
| `next.config.ts` | next-intl plugin integration |
| `src/proxy.ts` | Middleware (runs intl middleware first) |
| `messages/zh-TW.json` | Traditional Chinese translations |
| `messages/en.json` | English translations |

## Configuration

No special environment variables are needed for i18n. The locale configuration is entirely in code.

## Common Tasks

### Adding a new language

1. Add the locale code to the `locales` array in `src/i18n/routing.ts`:
   ```typescript
   locales: ["zh-TW", "en", "ja"],
   ```
2. Create a new translation file `messages/ja.json` with the same key structure as `en.json`.
3. Add the locale as a valid option in the profile update schema in `src/app/api/profile/route.ts`:
   ```typescript
   locale: z.enum(["zh-TW", "en", "ja"]).optional(),
   ```
4. Update the `buildSystemPrompt` function in `src/lib/chat-prompt.ts` to handle the new locale's language name.
5. Update the `generateMedicationPurpose` function in `src/lib/drugs.ts` to include a fallback string for the new locale.

### Adding new translation keys

1. Add the key to both `messages/en.json` and `messages/zh-TW.json`.
2. Use `useTranslations("namespace")` in the component to access the key:
   ```typescript
   const t = useTranslations("medications")
   return <p>{t("newKey")}</p>
   ```
3. For interpolated values, use the `{variable}` syntax in the JSON and pass values:
   ```json
   { "greeting": "Hello, {name}!" }
   ```
   ```typescript
   t("greeting", { name: "Alice" })
   ```

### Changing the default locale

Update `defaultLocale` in `src/i18n/routing.ts`. The `localePrefix: "as-needed"` strategy will then apply no prefix for the new default and add a prefix for all other locales.
